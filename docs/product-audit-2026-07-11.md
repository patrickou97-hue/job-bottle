# 拾星产品审计与任务型信息架构

审计日期：2026-07-11

## 当前产品定位

拾星是一款面向中文秋招学生的求职行动工作台，连接岗位发现、简历准备、投递跟进和结果复盘，并用星瓶保留有产品含义的可视化反馈。

## 核心用户

| 用户 | 场景 | 核心需求 |
| --- | --- | --- |
| 第一次参加秋招的学生 | 不知道先准备什么 | 快速建立求职目标，得到明确的第一步 |
| 已有方向、尚未记录岗位的学生 | 正在多个渠道找岗位 | 集中筛选岗位，判断是否值得投入时间 |
| 已收藏多个岗位的学生 | 截止日期和准备顺序混乱 | 比较机会，确定优先级，绑定合适简历 |
| 已进入笔试或面试的学生 | 同时跟进多个流程 | 记录节点、下一步动作和复盘，避免遗漏 |
| 手机端用户 | 在通勤或收到通知后快速更新 | 用较少步骤完成收藏、状态更新和备注 |

## 当前产品主链路

当前真实路径是：`/` 模块星图 -> `/explore` 岗位池 -> `/jobs/[id]` 岗位详情或直接打开官网 -> 创建 `opened` 投递记录 -> 返回后确认 `applied` -> `/my` 更新投递阶段 -> `/resume` 手动复制并关联岗位 -> `/bottle` 查看星瓶与分享。

这条链路已经能闭合，但“收录岗位”和“开始投递”在数据上共用 `opened`，岗位详情与简历定制没有直达动作，登录后的首页也不能回答“今天先做什么”。

## 当前主要问题

### P0：阻碍核心任务

| 页面 | 用户问题 | 原因 | 建议修改 | 预期效果 |
| --- | --- | --- | --- | --- |
| `/explore`、`/jobs/[id]`、`/my` | 用户无法区分“收藏待评估”和“已经开始准备” | `user_applications.status` 只有 `opened`，收藏、浏览和待准备被压成同一状态 | 新增兼容旧值的候选阶段或独立收藏属性，并把创建记录与打开官网拆开 | 用户可以先收藏和排序，再决定投递，不会产生虚假的投递记录 |
| `/jobs/[id]` | 详情页只能看基础元数据和自由备注，不能完成是否值得投的判断 | `jobs` 缺少职责、必须条件、优先条件等结构化字段；匹配仅使用偏好关键词 | 先按现有备注做清晰分区；后续 migration 增加结构化 JD 字段和可解释匹配结果 | 详情页从信息展示变成决策页面 |
| `/jobs/[id]` -> `/resume` | 针对岗位准备简历需要离开详情、手动复制、再手动关联 | 详情页没有携带岗位上下文，简历页没有接收岗位目标的入口 | 增加“针对该岗位准备简历”，由简历页创建或打开已绑定版本 | 岗位与简历版本真正形成连续流程 |
| `/my` 详情抽屉 | 无法记录完整投递渠道、账号、联系人、跟进时间和节点时间线 | 数据表只有状态、面试轮次和备注；`status_history` 已记录状态变化但前端未读取 | 先展示真实状态历史；后续 migration 增加结构化投递详情字段 | 用户能长期维护投递，而不是只改一个状态 |

### P1：明显降低效率或理解

| 页面 | 用户问题 | 原因 | 建议修改 | 预期效果 |
| --- | --- | --- | --- | --- |
| `/` | 新用户能看到模块，但不知道第一步；登录用户看不到今日任务 | 首页始终是模块星图，未区分访客介绍与登录后工作台 | 访客保留品牌入口并突出“找岗位”；登录后优先进入任务工作台，星图降为探索入口 | 首屏直接回答当前阶段和下一步 |
| 全局导航 | “岗位池、投递、资料、星瓶”按功能并列，星瓶与投递资产重复 | 导航沿用功能模块，而非用户任务 | 收敛为找岗位、投递管理、简历、求职交流、我的；星瓶进入“我的”资产区 | 降低导航理解成本，保留品牌概念但不抢主链路 |
| `/my` | 已有看板和星图，但缺少适合搜索、批量浏览的列表视图 | 当前视图枚举只有 `board` 和 `map` | 增加列表视图，移动端默认列表，看板用于阶段比较 | 高密度场景和手机端更易扫描 |
| `/my` | “今日待办”是通用状态文案，不是真正基于时间的提醒 | 当前任务只按状态与备注排序，没有 next action date | 明确标注为“优先处理”；增加 `next_action_at` 后再升级为今日提醒 | 避免伪造时间语义，提高信任感 |
| `/resume` | 模板选择仍占据版本管理之前的显著位置；列表缺少使用次数、绑定岗位名、完整度 | 当前模型只保存模板和 `linked_job_id`，投递记录未保存 `resume_id` | 将岗位版本入口前置；后续增加投递到简历的引用与使用统计 | 用户围绕内容和版本工作，而不是围绕模板工作 |
| `/bottle` | 星瓶展示统计和分享，但缺少候选岗位排序、截止日期和状态筛选 | 页面是可视化成果页，不是候选管理页 | 保留星瓶为结果反馈；候选管理放在找岗位下的线性列表，不把筛选塞进瓶子 | 品牌体验与高频管理各司其职 |
| 登录/注册 | 注册一次收集较多资料，但完成后缺少明确初始化进度 | 注册表单与资料页有字段，缺少短流程反馈 | 注册只保留账号和少量目标；登录后用三步检查清单补齐偏好与基础简历 | 更快进入产品，同时保持资料可完善 |

### P2：品质、信任与细节

| 页面 | 用户问题 | 原因 | 建议修改 | 预期效果 |
| --- | --- | --- | --- | --- |
| `/jobs/[id]` | “星球详情”、大量圆角面板弱化专业求职语境 | 品牌隐喻进入了高频决策页 | 改为岗位详情，使用分隔线和开放布局，仅保留轻量品牌色 | 更可信、更接近系统工具的信息密度 |
| 多个页面 | 错误信息常写“稍后再试”，没有总是说明输入是否保留 | 错误状态未形成统一文案协议 | 统一“发生了什么 + 内容是否保留 + 下一步” | 用户敢于持续录入重要信息 |
| 移动端 | 依赖汉堡菜单，投递看板四列只能纵向堆叠 | 桌面信息架构直接响应式压缩 | 后续使用首页、岗位、投递、我的四项底部导航；投递默认列表 | 高频动作单手可达，避免把桌面看板硬塞进手机 |
| 简历预览 | 原网页预览在窄屏只缩纸张、不缩字体，且 PDF 会动态压缩字号 | HTML 和 jsPDF 是两套排版器 | 预览与 PDF 共用 A4 坐标、字体和分页操作 | 所见即所得，桌面和手机只改变显示比例 |

## 新的信息架构

### 一级导航

1. 找岗位：岗位列表、岗位详情、候选岗位。
2. 投递管理：优先处理、列表、看板、投递详情和复盘。
3. 简历：基础简历、方向版本、岗位定制版本。
4. 求职交流：经验库与讨论。
5. 我的：求职档案、偏好、星瓶、账号与反馈。

首页由品牌入口改为上下文入口：未登录展示产品价值和“开始找岗位”；已登录展示求职工作台。桌面品牌标识仍可返回首页。移动端最终收敛为首页、岗位、投递、我的四项，简历从首页任务和“我的”进入。

### 模块关系

```text
求职档案 -> 匹配解释 -> 候选岗位 -> 岗位定制简历
         -> 投递记录 -> 状态时间线 -> 下一步动作 -> 复盘
候选岗位与投递结果 -> 星瓶可视化反馈
```

## 核心用户流程

### 新用户

访问首页 -> 登录/注册 -> 选择地区与岗位方向 -> 创建或上传基础简历 -> 浏览第一批岗位 -> 收藏第一个候选岗位。

### 岗位发现

筛选岗位 -> 打开详情 -> 查看条件与匹配依据 -> 收藏 -> 设置优先级 -> 开始准备。

### 简历准备

岗位详情 -> 针对岗位准备简历 -> 选择基础版本 -> 创建绑定副本 -> 修改内容 -> 保存 -> 返回投递计划。

### 投递管理

开始准备 -> 打开官网 -> 确认已投递 -> 记录日期与渠道 -> 更新笔试/面试 -> 设置下一步 -> 记录结果。

### 复盘

投递管理 -> 筛选长期未更新和已结束记录 -> 查看状态时间线 -> 记录原因与经验 -> 调整偏好和后续策略。

## 数据判断

当前状态枚举在前端和数据库中是一致的：`opened`、`applied`、`written_test`、`first_round`、`second_round`、`final_round`、`offer`、`rejected`、`withdrawn`。问题不是多套技术枚举，而是 `opened` 承担了浏览、收藏和待准备三个产品含义。

第一阶段不应把渠道、联系人、提醒时间或优先级继续塞进 `progress_note`。后续 migration 应以 nullable 字段兼容旧记录，并保留旧状态映射。

## 实施边界

本轮先完成：A4 简历预览与 PDF 同源；岗位详情到岗位简历版本的直达入口；导航命名收敛；真实错误和空状态修正；smoke 防回归。

需要数据库迁移后完成：独立候选/收藏阶段、优先级、投递渠道与账号、联系人、下一步时间、投递使用的简历 ID、结构化 JD、状态时间线前端读取和完整埋点漏斗。
